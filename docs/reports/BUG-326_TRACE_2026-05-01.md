# BUG-326 — protected_identity gate bypass trace

## What we know (the bug)

Janice's memoir export on 2026-05-01 live test shows two protected identity fields polluted with garbage:

- `personal.placeOfBirth`: "My dad worked nights at the aluminum plant.." (narrator's prior turn text)
- `personal.birthOrder`: "I just told you something you ignored." (resistance/complaint phrase)

Expected values were Spokane, Washington and unset (or "youngest").

Both fields are marked as `protected_identity` in the schema, and BUG-312 (2026-04-28) was supposed to block untrusted sources from writing to them — routing instead to `suggest_only` candidates for operator review.

## What BUG-312 does (the gate)

**File:** `ui/js/projection-sync.js`  
**Lines:** 91–127

The BUG-312 gate (lines 91–127) is a Phase G integrity check in `projectValue()`:

```javascript
// ── Phase G: PROTECTED IDENTITY CHECK ──
// Protected identity fields (fullName, DOB, placeOfBirth, etc.) can ONLY
// be written by trusted sources (human_edit / preload / profile_hydrate).
// Any untrusted source (interview / backend_extract / projection /
// backend_correction) ALWAYS routes through suggest_only — even on the
// first write to a blank field.
var PM = window.LorevoxProjectionMap;
if (PM && PM.isProtectedIdentity && PM.isProtectedIdentity(fieldPath)) {
  if (!_isTrustedSource(source)) {
    _logSync(fieldPath, "blocked_protected_identity", existing && existing.value, value);
    // Route to suggestion instead of direct write — operator approves
    // via Shadow Review.
    _syncSuggestOnly(fieldPath, value, confidence);
    console.warn("[projection-sync] ⛔ Protected identity write from untrusted source — routed to candidates: " +
      fieldPath + " value=" + JSON.stringify(value).slice(0, 80) + " source=" + source);
    return false;
  }
  // ... trusted source path continues
}
```

**Trusted source whitelist (line 180–181):**

```javascript
function _isTrustedSource(source) {
  return source === "human_edit" || source === "preload" || source === "profile_hydrate";
}
```

The gate should have caught writes from `"interview"` and `"backend_extract"` sources and routed them to `suggest_only`.

## Most likely culprit

**The BUG-312 gate itself is working correctly** — it IS blocking the `backend_extract` writes to protected identity fields. The real problem is that the **backend extractor is emitting hallucinated/malformed field paths with garbage values** that, even when routed to `suggest_only`, end up in the candidates queue and are later **approved by the operator or accepted via the acceptance flow** without realizing they are garbage.

However, the observed values in the memoir export suggest the garbage **was written directly to truth** (placeOfBirth and birthOrder), not left as pending suggestions. This means:

1. **Either the BUG-312 gate is not firing** (the isProtectedIdentity map isn't marking these fields correctly), or
2. **The extraction results are bypassing `projectValue()` entirely** and writing directly to the bio builder or projection, or
3. **A different code path** (perhaps questionnaire_first auto-save or bio-builder PUT operations) is writing these values without going through the projection-sync gate.

Looking at session-loop.js `_saveBBAnswer()` behavior: when questionnaire_first asks for a field and the narrator responds, the answer is saved directly to the Bio Builder questionnaire via `PUT /api/bio-builder/questionnaire`, which does NOT go through projection-sync. If the extraction happened to run AFTER the direct save, the extracted values would be routed through projectValue() and properly gated. But if the extraction ran CONCURRENTLY or earlier, and somehow its source signature got mis-stamped as a trusted source, those values could have overwritten the good ones.

**Most probable scenario:** The backend extractor emitted `personal.placeOfBirth` and `personal.birthOrder` fields with narrative-fragment values (pulled from prior turns or context), these were assigned `source: "backend_extract"`, the interview.js code at L1430 called `projectValue()` with this untrusted source, but the BUG-312 gate **either didn't fire or wasn't properly protecting these specific fields** due to a schema mapping issue or race condition in the projection state.

## Recommended fix scope

1. **CRITICAL: Audit the isProtectedIdentity mapping** in `projection-map.js` to confirm `personal.placeOfBirth` and `personal.birthOrder` are included in the protected-identity list. If they're missing, add them immediately.

2. **Add console logging to BUG-312 gate** to capture whenever it blocks a protected-identity write. The logs should appear in the browser console as `[projection-sync] ⛔ Protected identity write from untrusted source...`. If Janice's session logs don't show these warnings for placeOfBirth and birthOrder, the gate is not firing.

3. **Audit backend extractor for hallucinated placeOfBirth/birthOrder paths.** The backend (`server/code/api/routers/extract.py`) is emitting `personal.placeOfBirth` with values like "My dad worked nights at the aluminum plant.." which is clearly a Type C binding failure. The BUG-EX-PLACE-LASTNAME-01 guard (added 2026-04-29) only covers `.lastName` / `.maidenName` / `.middleName`, not `.placeOfBirth`. Consider extending that guard or adding a separate guard for place-of-birth values that look like narrative fragments (contain relative pronouns, past-tense narrative, etc.).

4. **Trace the questionnaire_first flow in session-loop.js** — ensure that when `_saveBBAnswer()` is called, it's not bypassing projection-sync or stamping sources incorrectly. The `qf-identity-rescue` writes in session-loop.js use `source: "interview"` which is correct (untrusted), but confirm the bio-builder PUT path at `_saveBBAnswer()` doesn't have a secondary write path that uses a trusted source.

5. **Check for race conditions** between backend extraction (async, fires ~1s after narrator turn) and questionnaire_first auto-save (synchronous on narrator turn). If auto-save writes a blank field to the BB questionnaire, and then extraction tries to project a value to the same field, the order matters. The confidence gate (L129–140 in projection-sync.js) might allow lower-confidence interview extractions to overwrite higher-confidence BB writes if the timestamps are close.
