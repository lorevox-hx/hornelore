# WO-13 Phase 9 — Plan

**Session orientation layer + banner state machine + operator clarity layer**

Author: Chris + Claude
Date drafted: 2026-04-11
Status: Draft for comparison. Do NOT start execution until Phase 8 rollout
(B → A → C) is stable on Kent.

---

## 0. One-paragraph purpose

Phase 8 made the read seam correct: the profile endpoint returns
promoted-truth data under the flag, legacy passthrough under the flag,
and never corrupts either. Phase 9 makes the read seam *legible* — to
the LLM that composes Lori's responses, to the operator who runs the
review queue, and to the end user who is looking at a profile in a
mixed state. None of Phase 9 writes to the truth pipeline. None of
Phase 9 changes what the profile endpoint returns. Phase 9 is pure
presentation: session payload, prompt composition, UI banner, dev-mode
diagnostics. If Phase 8 is "the database knows the truth," Phase 9 is
"everyone looking at the database understands what kind of truth it is."

## 1. Scope

**In scope**

- A session orientation block — ephemeral, per-session, injected into
  the prompt, never persisted to any truth table.
- A four-state banner that tells the operator the current Memory Truth
  lifecycle state of the loaded narrator.
- A dev-mode-only diagnostics surface that exposes `source` and
  per-field provenance (hover tooltip on each profile field).
- One new prohibition added to the WO-13 invariants: age is never
  extracted from conversation, only computed from an approved DOB.

**Out of scope**

- Any change to `family_truth_notes`, `family_truth_rows`, or
  `family_truth_promoted`.
- Any change to `build_profile_from_promoted` or the profile endpoint.
- Any change to the flag gating. `HORNELORE_TRUTH_V2_PROFILE` continues
  to control exactly what it controls today.
- Any change to `ft_backfill_from_profile_json`. If backfill behaviour
  needs to change, that's a separate work order.
- Auto-generated biography content of any kind.
- LLM-side precedence instructions ("use X first, Y as tentative").
  Precedence is a code concern, not a prompt concern.

## 2. Architectural boundaries (the things Phase 9 must never do)

These are load-bearing. Violating any one of them re-introduces the
class of bugs WO-13 was designed to prevent.

1. **No synthetic truth rows at session start.** The session orientation
   block never writes to `family_truth_notes` or `family_truth_rows`.
   Its entire existence is confined to the session payload.
2. **No startup narrative text in `profile_json`.** The banner state is
   computed from existing tables, not stored.
3. **No current-date-as-biography.** System time is computed in the
   prompt composer, never persisted.
4. **No LLM precedence instructions.** The prompt never says "use
   promoted truth first" — it receives an already-filtered profile and
   an already-filtered rolling summary, and does not know that filtering
   happened.
5. **No age extraction from conversation.** Lori does not capture "I am
   82" as a fact. Age is derived from `basics.dob` in the composer and
   passed in as ephemeral context.
6. **No end-user exposure of `source`.** The raw source tag
   (`legacy` / `promoted_truth` / `legacy_fallback`) is a systems
   signal. End users see human-language equivalents through the banner
   and tooltips, never the tag itself.

## 3. The three surfaces

### 3.1 Session orientation block (prompt-side)

**What it is.** A short string, computed at narrator load, that is
prepended to the system prompt for the duration of the session. It
tells the LLM *what kind of conversation this is* and nothing else. It
does not describe data precedence, data freshness, or data governance —
those are the prompt composer's job, not the LLM's.

**Where it lives.**

- Generated by a new helper, `build_session_orientation(person_id)`,
  probably in `server/code/api/session.py` (needs confirmation via code
  survey before execution).
- Stored on the `session_payload` dict under key `orientation` for the
  duration of the session. The existing `POST /api/session/put`
  endpoint already round-trips arbitrary keys, so no schema change.
- Read by the prompt composer (wherever that lives — probably
  `server/code/api/chat.py`; another code-survey item). Inserted at the
  top of the system prompt, before profile facts.

**The two posture strings** (subject to copy review, but these are the
shapes).

*LIVE narrator:*

```
This narrator is sharing their life story from birth to the present.
Respond as a warm, attentive interviewer. Focus on helping them tell
their story clearly and naturally.
```

*REFERENCE narrator:*

```
This is a reference narrator. Treat their profile as curated and
read-only. Do not infer or generate new truth from session interaction.
```

**What the strings deliberately do not contain.**

- No "use approved truth first."
- No "treat unreviewed material as tentative."
- No "do not invent facts."
- No "use system date to frame the present."
- No per-field instructions about precedence, provenance, or trust.

All of those are either handled in code (precedence, facts-write
freeze, rolling-summary filter) or counterproductive (LLM hallucination
priming).

**Computed present-time context** is a separate dict key on the session
payload, not part of the orientation string. It contains:

- `now_iso` — system wall clock at session start (refreshed on each
  prompt compose, not at session start only).
- `narrator_age_years` — integer, computed as `(now - basics.dob)` if
  and only if `basics.dob` is both present *and* sourced from
  `promoted_truth` (not legacy). If DOB is legacy or missing, this key
  is absent from the payload and the composer does not inject an age
  statement into the prompt.

**Age computation rule, stated explicitly.** Age is never extracted
from conversation. Lori does not capture "I am 82" as a fact. The only
way `narrator_age_years` becomes populated is arithmetic on an approved
DOB. This is added to the WO-13 invariants and enforced in the
extraction code path (probably in `facts.py` and/or the rolling-summary
filter — code survey item).

### 3.2 Banner state machine (UI-side)

**What it is.** A small persistent element at the top of the profile
view that tells the operator which phase of the Memory Truth lifecycle
the current narrator is in. It is driven entirely by three inputs and
has exactly four states.

**Inputs.**

1. Does the narrator have *any* rows in `family_truth_rows`?
   (cheap COUNT query)
2. Does the narrator have *any* rows in `family_truth_promoted`?
   (cheap COUNT query)
3. Of the rows in `family_truth_rows`, how many are still
   `needs_verify` vs. terminal (`approve`, `approve_q`, `reject`,
   `source_only`)? (cheap GROUP BY)

All three are already computable from existing tables — no schema
change. They can be served by a new endpoint,
`GET /api/family-truth/state/{person_id}`, or folded into the existing
profile endpoint response as a `memory_truth_state` block. My
preference is the latter because it avoids a second round trip on
profile load, but this is a code-survey decision.

**The four states.**

| State | Trigger | Banner copy |
|---|---|---|
| **0 — Not set up** | `rows == 0 && promoted == 0` | "This narrator hasn't been set up for Memory Truth yet. Click *Backfill* to stage the existing profile for review." |
| **1 — Backfilled, nothing reviewed** | `rows > 0 && promoted == 0` | "N items are waiting for your review before they become approved truth. Until then, the profile shows legacy data." |
| **2 — Partial review** | `promoted > 0 && needs_verify > 0` | "N items approved, M still pending. The profile mixes reviewed and legacy data — hover any field to see its source." |
| **3 — Fully reviewed** | `promoted > 0 && needs_verify == 0` | (No banner, or a small confirmation chip: "Memory Truth: all fields reviewed.") |

State 3's "no banner" behaviour is deliberate — once a narrator is
fully reviewed, the operator shouldn't be nagged. The confirmation
chip is a consolation prize if usability testing shows state 3
narrators feel "abandoned" by the UI. Decide after A.

**Reference narrators** are a special case. They never get backfilled
and they should never be reviewed, so none of the four states apply.
They get a fixed banner: "Reference narrator — profile is curated and
read-only. No review queue." This is computed before the four-state
machine runs.

**Where it lives in the DOM.** New `<div id="wo13MemoryTruthBanner">`
at the top of the profile pane, inserted by a new function
`wo13RenderMemoryTruthBanner(state)` in a new file
`ui/js/wo13-banner.js`. The banner re-renders on:

- Initial profile load (after `lvxRefreshProfileFromServer`).
- Every successful promote (the refresh shim already fires — just
  have it re-render the banner too).
- Every backfill call (new code path).

**CSS.** Four state-specific classes — `wo13-banner-state-0` through
`-3` — plus `wo13-banner-reference`. Matches the existing Phase 6 CSS
convention (`wo13-status-approve`, etc.).

### 3.3 Operator clarity layer (dev-mode diagnostics)

**What it is.** A small collection of diagnostics visible only when the
operator has flipped a developer-mode switch. These exist to help *you*
(and future you, and anyone else who ever has to debug this system) —
not end users.

**The switch.** A new flag `HORNELORE_DEV_MODE` on the client side,
probably stored in `localStorage.lorevox_dev_mode`. Toggled via a
hidden keyboard shortcut (suggestion: `Ctrl+Shift+D` matches no existing
Hornelore binding — needs verification). When off, everything in this
section is hidden.

**What dev mode shows.**

1. **`source` tag in the banner.** The banner already computes state
   from row counts. In dev mode, it additionally shows the raw
   `source` value from the profile endpoint response
   (`legacy` / `promoted_truth` / `legacy_fallback`) as a small pill on
   the right side of the banner. If the operator ever sees
   `legacy_fallback` in production, that's the signal the builder
   crashed and the server log needs checking.
2. **Per-field provenance tooltip.** Hovering any field in the profile
   form shows a tooltip like: "Reviewed by chris, 2026-04-11, from
   chat source sess1:0." For legacy-passthrough fields, the tooltip
   says: "Legacy profile data — not yet reviewed." This requires
   exposing one new piece of information from the builder: for each
   basics key, is it promoted or legacy? The cleanest way to do this
   is an additive sidecar `basics._provenance: {<key>: "promoted" |
   "legacy"}` written by `build_profile_from_promoted`. This is a
   small, additive, backwards-compatible extension to the builder and
   does not change its precedence rules.
3. **Debug panel.** A collapsed panel under the banner (opens to a
   pre-formatted JSON block) showing the full profile endpoint
   response: `{person_id, profile, source, updated_at,
   memory_truth_state}`. Useful for bug reports and for verifying the
   server actually returned what you expected.

**End users never see any of this.** The same profile in non-dev-mode
shows the banner (with no `source` pill, no provenance tooltip, no
debug panel) and the fields normally.

## 4. Design tensions and decisions

These are the choices I want to flag as *decided* so they don't get
relitigated during execution.

1. **Orientation is a posture statement, not a precedence instruction.**
   We do not tell the LLM how to rank data sources. The prompt composer
   does that before the LLM ever sees anything. This was the single
   most important correction from the pushback round and it is
   non-negotiable.
2. **"Do not invent facts" is removed.** It is redundant with Phase 4/5/7
   structural defenses and risks nocebo-priming the model. It does not
   appear in either posture string.
3. **Banner copy is state-specific, not generic.** A single "Memory
   Truth Mode Active" banner lies in state 1. Each state has its own
   wording that tells the operator something true about *this*
   narrator's *current* lifecycle position.
4. **`source` is a dev-mode signal, not an end-user label.** End users
   see "approved," "pending review," or nothing — never "legacy" or
   "legacy_fallback." Those words mean something to us; they mean
   something scary and unclear to family members using Hornelore to
   preserve a relative's memoir.
5. **`narrator_age_years` requires an approved DOB.** Legacy DOB alone
   does not populate the age field in the session payload. This is
   strict — if you want age, you have to review DOB first. It's a
   small pressure in favor of doing the review, and it prevents the
   age field from drifting based on unreviewed data.
6. **Banner renders on mount, refresh shim, and backfill call.** Not
   on every state change, not on every keystroke, not on a timer.
   Three triggers, explicit list, no exceptions.
7. **The four-state machine is computed on the server, not the
   client.** The client receives `memory_truth_state: {state: 0|1|2|3,
   counts: {rows, promoted, needs_verify}}` as part of the profile
   endpoint response and renders it. Computing on the client would
   require three extra round trips to count rows, and the state
   machine logic would need to be duplicated in two languages.

## 5. Implementation order

This is a guess at order-of-operations. Final ordering is a code-survey
decision at Phase 9 kickoff.

1. Server: add `memory_truth_state` computation to `api_get_profile`.
   New helper `db.ft_lifecycle_state(person_id)` that runs three cheap
   counts and returns `{state, counts}`. Ship this first because
   everything else depends on it.
2. Server: extend `build_profile_from_promoted` to emit
   `basics._provenance` sidecar. Small, additive, backwards compatible.
3. Server: add `build_session_orientation(person_id)` and wire it into
   the session payload at session start. Decide whether it fires on
   every `session_put` or only on first load (leaning toward first
   load — it's per-session, not per-message).
4. Server: find the prompt composer, insert the orientation string at
   the top of the system prompt, and compute `narrator_age_years` +
   `now_iso` at compose time. This is the riskiest step because the
   prompt composer is a stateful thing and changes here will affect
   every session. Should have its own test harness.
5. Client: add `wo13-banner.js` + CSS. Render on profile load and
   refresh.
6. Client: extend `lvxRefreshProfileFromServer` to also re-render the
   banner (one-line addition).
7. Client: add dev-mode toggle, `source` pill, provenance tooltips,
   debug panel. All four of these are dev-mode-gated and can ship
   together.
8. Add the age-extraction prohibition to the facts/rolling-summary
   filter code and write a regression test that a message containing
   "I'm 82 years old" does not produce any row in
   `family_truth_rows` or any entry in the rolling summary that carries
   an age claim.

## 6. Test strategy

**Server-side, pure Python (`run_phase9_server_test.py`).**

- `ft_lifecycle_state` returns state 0/1/2/3 for fabricated row
  patterns (one narrator per state).
- Reference narrator returns a special "reference" state, not 0-3.
- `build_session_orientation` returns the right posture string for
  LIVE vs REFERENCE.
- `build_profile_from_promoted` emits `_provenance` sidecar with
  correct per-key tags (promoted for the 2 Ada promoted fields,
  legacy for the 3 that weren't promoted).
- `narrator_age_years` is populated iff DOB is promoted.
- `narrator_age_years` is *not* populated when DOB exists only in
  legacy.
- Age extraction regression: a message "I'm 82" fed through the
  extraction pipeline produces zero age-related rows.

**Client-side, jsdom harness (`run_phase9_ui_test.js`).**

- Banner renders state 0 / 1 / 2 / 3 / reference for five fabricated
  profile endpoint responses.
- Banner copy matches expected text for each state.
- Dev-mode toggle hides/shows `source` pill, provenance tooltips, and
  debug panel.
- Hovering a promoted field shows the right tooltip; hovering a legacy
  field shows a different tooltip.
- Clicking promote fires the refresh shim and the banner state
  transitions from 1 to 2 (or 2 to 3 if that was the last pending row).

**Regression.** Full Phase 4/5/6/7/8 suites must stay green. No
exceptions.

## 7. Rollout recipe

1. Land Phase 9 with all dev-mode features OFF. The banner is on by
   default (it's informational, not gated) but dev-mode diagnostics
   require the flag.
2. Verify on Kent. His state will be 2 or 3 depending on how far the
   Phase 8 rollout got. Banner copy should match.
3. Create a fresh narrator and verify state 0 → state 1 (after
   backfill) → state 2 (after first promote) → state 3 (after
   completing review).
4. Flip dev mode on. Verify provenance tooltips and debug panel.
5. Start a real session as Kent. Verify the orientation block reaches
   the prompt composer and that Lori's opening line has the right
   posture (warm, interviewer-style, does not mention "approved
   truth" or "tentative material" or any systems language).

## 8. Known unknowns

These will need resolution during Phase 9 kickoff — don't assume I
have them pinned down yet.

- Where the prompt composer actually lives. I've guessed `chat.py` but
  this needs grep confirmation.
- How `session_payload` is currently being round-tripped. I know
  `/api/session/put` exists and that `state.chat.conv_id` is the key;
  I don't yet know the schema on the server side.
- Whether the existing rolling-summary filter already catches age
  claims or whether we need to add a new rule. If it already does,
  the age-extraction test is a regression guard; if not, it's a new
  feature.
- Whether there's a clean way to expose `basics._provenance` to the
  client-side tooltip code without adding a third read path to the
  profile form hydration logic. Probably yes via `state.profile`.
- What copy review (if any) the banner strings need. I've written them
  as a single developer; they may want softer phrasing or may need to
  respect a product voice that I don't know about.
- Whether the `Ctrl+Shift+D` dev-mode binding conflicts with any
  existing Hornelore keyboard shortcut. Grep first, decide later.

## 9. Explicit non-goals (restated for emphasis)

Phase 9 does not:

- Write to any `family_truth_*` table.
- Change `build_profile_from_promoted`'s precedence rules.
- Change the flag gating.
- Add a new promotable status.
- Auto-generate any biography content.
- Instruct the LLM how to rank data sources.
- Expose raw `source` tags to end users.
- Capture age claims from conversation.
- Build a memoir composer (that's a different work order entirely).
- Touch the legacy `profile_json` write path.

## 10. Exit criteria

Phase 9 is done when all of the following are true:

1. `run_phase9_server_test.py` passes with all assertions green.
2. `run_phase9_ui_test.js` passes with all assertions green.
3. Full Phase 4/5/6/7/8 regression is green.
4. Kent's profile shows the correct banner state for his actual
   lifecycle position (dry run + live verification).
5. A fresh narrator walks through states 0 → 1 → 2 → 3 cleanly with
   the banner copy transitioning correctly at each boundary.
6. Dev mode can be toggled on and off, and all three dev-mode
   features (source pill, provenance tooltip, debug panel) behave
   correctly in both states.
7. A real chat session with Kent shows the LLM adopting the correct
   posture from the orientation block — no "approved truth" language
   in Lori's responses, no precedence talk, no age guessing.
8. The age-extraction regression test proves that "I'm 82" does not
   produce any row in `family_truth_rows`.
9. `PHASE9_PROOF.md` is written and committed following the same
   shape as Phase 7 and Phase 8 proof packets.

## 11. Things to decide at Phase 9 kickoff (not now)

- Exact location of the prompt composer.
- Exact shape of `session_payload.orientation` (string vs dict).
- Whether `memory_truth_state` ships as part of the profile endpoint
  response or as a separate endpoint.
- Dev-mode keyboard binding.
- Banner copy review (possibly with a second set of eyes).
- Whether reference narrators get a "state" field at all or just a
  flag.
- Whether `_provenance` ships as `basics._provenance` (sidecar key on
  basics) or `profile._provenance` (top-level alongside basics). The
  former is consistent with `_qualifications`; the latter is slightly
  cleaner if we ever want kinship provenance too.
